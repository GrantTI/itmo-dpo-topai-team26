### Задание №1. «Найди 5 ошибок в C4-диаграмме»

**Формулировка:**  
Перед вами диаграмма (уровень Container), которую нарисовал стажёр для интернет-магазина:

```
[Пользователь] → (HTTPS) → [Nginx:80] → (HTTP) → [FastAPI:8000] → (SQL) → [PostgreSQL:5432]
                                                              ↓
                                                         [Vue.js SPA]
                                                              ↓
                                                         (читает статику)
```

**Задание:**  
1. Найдите **минимум 5 ошибок** в этой диаграмме с точки зрения нотации C4 и здравого смысла.  
2. Перерисуйте диаграмму **исправленно** в виде текстового описания (или от руки).  
3. Подпишите **одну** стрелку, указав на ней: протокол, порт и направление данных (кто инициирует запрос).

---

**Пример верного ответа:**

> **Найденные ошибки:**
> 1. **Nginx слушает порт 80**, но пользователь идёт по HTTPS — несоответствие (должен быть 443).  
> 2. **Vue.js SPA** находится на одной линии с FastAPI, но это не контейнер (это статика, которую отдаёт Nginx).  
> 3. **Vue.js не может "читать статику"** — он сам является статикой; правильно: Nginx отдаёт файлы браузеру.  
> 4. **Стрелка от FastAPI к Vue.js** — Backend не общается с Frontend'ом напрямую (только через HTTP-запросы от браузера).  
> 5. **Отсутствует легенда** — непонятно, что означают цвета и типы линий.  
> 6. *(Бонус)* На C4-диаграмме не пишут IP-адреса и порты внутри блоков, их выносят на стрелки.

> **Исправленная версия (текстовое описание):**
> - **Пользователь** (человечек) → **Nginx** (порт 443, HTTPS) — запрос статики.  
> - **Nginx** → браузер отдаёт файлы `index.html`, `app.js`, `css` (HTTP/2).  
> - **Пользователь** → **Nginx** (порт 443, HTTPS) — API-запросы на `/api/*`.  
> - **Nginx** → **FastAPI** (порт 8000, HTTP) — проксирование.  
> - **FastAPI** → **PostgreSQL** (порт 5432, TCP) — чтение/запись данных.  
> - **Nginx** отдаёт статику, **FastAPI** — только JSON-ответы.

> **Пример подписи стрелки:**  
> `Браузер → Nginx: "Запрос /api/orders" (HTTPS, TCP/443)`

### Задание №4. «Раздели монолит на два экземпляра с балансировкой»

**Формулировка:**  
Вам дана рабочая схема интернет-магазина:

```
[Покупатель] → (HTTPS) → [Nginx:443] → (HTTP) → [FastAPI:8000] → (TCP) → [PostgreSQL:5432]
                                                      ↓
                                                 отдаёт статику (Vue.js)
```

При чёрной пятнице нагрузка выросла в 10 раз. FastAPI начал падать по CPU.  
Решение: **запустить 3 экземпляра FastAPI** за одним Nginx. Nginx должен распределять запросы между ними (round-robin).  

**Задание:**  
1. Перерисуйте диаграмму, добавив **3 контейнера FastAPI**.  
2. Покажите на стрелках, как Nginx распределяет запросы (напишите `upstream backend { ... }` в псевдокоде).  
3. Объясните **одним предложением**, почему PostgreSQL остался один, а не 3 копии.  
4. Добавьте на схему **Health Check** (пунктирная стрелка от Nginx к каждому FastAPI) — для проверки, жив ли экземпляр.

---

**Пример верного ответа:**

> **Схема контейнеров:**
> - **Nginx** (порт 443) — балансировщик + прокси.  
> - **FastAPI-1**, **FastAPI-2**, **FastAPI-3** — три независимых контейнера на портах 8001, 8002, 8003.  
> - **PostgreSQL** (один) — общее хранилище.  
> - **Vue.js** — статика, отдаётся через Nginx.

> **Upstream-блок (псевдокод):**
> ```nginx
> upstream fastapi_cluster {
>     server fastapi-1:8001;
>     server fastapi-2:8002;
>     server fastapi-3:8003;
> }
> 
> location /api/ {
>     proxy_pass http://fastapi_cluster/;
> }
> ```

> **Health Check (пунктир):**  
> `Nginx → FastAPI-1 (GET /health)`, `Nginx → FastAPI-2 (GET /health)`, `Nginx → FastAPI-3 (GET /health)` — если не отвечает 200, Nginx исключает экземпляр из ротации.

> **Объяснение:** *"PostgreSQL остался один, потому что это состояние (stateful-сервис), и синхронизация между тремя БД потребовала бы сложного репликационного кластера, что выходит за рамки текущей задачи."*

---

**Критерии оценивания (max 10 баллов):**

| Критерий | Баллы | Пояснение |
|----------|-------|-----------|
| **Наличие 3 экземпляров FastAPI** | 2 | Нарисованы три отдельных контейнера (не один с пометкой "x3") |
| **Upstream-конфигурация** | 3 | Показан блок с перечислением всех трёх серверов |

### Задание №6. «Интеграция с внешним платёжным шлюзом»

**Формулировка:**  
У вас есть базовая схема онлайн-кинотеатра:

```
[Пользователь] → [Nginx] → [FastAPI] → [PostgreSQL]
                                 ↓
                            [Redis] (кэш фильмов)
```

Новая функциональность: **оплата подписки** через внешний сервис **Stripe** (или любой платёжный шлюз).  
Алгоритм:  
1. Пользователь нажимает "Купить подписку".  
2. FastAPI создаёт платёжную сессию и передаёт пользователя на Stripe.  
3. Stripe присылает **Webhook** (POST-запрос) на ваш бэкенд после успешной оплаты.  
4. FastAPI обновляет статус подписки в PostgreSQL.

**Задание:**  
1. Нарисуйте схему, добавив **внешнюю систему** (Stripe) как отдельный актор или систему (согласно C4, это внешний софт).  
2. Покажите **два пути** трафика:  
   - **Синхронный** (пользователь → FastAPI → Stripe → пользователь).  
   - **Асинхронный** (Stripe → FastAPI — Webhook).  
3. Подпишите стрелки, указав, какой метод HTTP используется (`POST /create-payment`, `POST /webhook`).  
4. Добавьте **Redis** в цепочку: FastAPI должен сохранять `payment_id` в Redis до подтверждения Webhook'а.

---

**Пример верного ответа:**

> **Добавленные элементы:**  
> - **Stripe** (внешняя система, рисуется серым или пунктирной рамкой).  
> - **Redis** (уже был, теперь добавлена стрелка от FastAPI).

> **Потоки:**
> 1. **Пользователь → FastAPI** — `POST /create-payment` (HTTPS).  
> 2. **FastAPI → Redis** — сохраняет `payment_id: user_id, status=pending` (TCP).  
> 3. **FastAPI → Stripe** — запрос на создание сессии (HTTPS, POST).  
> 4. **Stripe → Пользователь** — редирект на платёжную страницу (внешний).  
> 5. **Stripe → FastAPI** — `POST /webhook` (HTTPS) — уведомление об успешной оплате.  
> 6. **FastAPI → Redis** — обновляет статус на `paid`.  
> 7. **FastAPI → PostgreSQL** — обновляет подписку пользователя.

> **Подписи стрелок:**  
> - `FastAPI → Stripe: "POST /create-payment" (HTTPS, порт 443)`  
> - `Stripe → FastAPI: "POST /webhook" (HTTPS, порт 443)`

> **Важное замечание:** Webhook должен иметь **публичный URL** (не localhost), поэтому Nginx должен проксировать `/webhook` на FastAPI без авторизации (или с проверкой подписи Stripe).

---

**Критерии оценивания (max 10 баллов):**

| Критерий | Баллы | Пояснение |
|----------|-------|-----------|
| **Stripe как внешняя система** | 2 | Отдельный блок, выделен цветом или стилем (не как внутренний контейнер) |
| **Два потока (синхронный/асинхронный)** | 3 | Показаны и подписаны оба маршрута (редирект и webhook) |
| **Методы HTTP подписаны** | 2 | Указаны `POST /create-payment` и `POST /webhook` |
| **Redis в цепочке** | 2 | FastAPI сохраняет и читает из Redis до/после вебхука |
