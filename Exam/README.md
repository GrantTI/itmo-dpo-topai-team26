## 1. Общие положения

Контрольно-измерительные материалы (КИМ) предназначены для проведения дифференцированного зачета по дисциплине, направленной на освоение принципов разработки, интеграции и развертывания веб-приложения с моделью машинного обучения.

Дифференцированный зачет является формой промежуточной аттестации и служит для оценки степени освоения обучающимися содержания дисциплины в целом.

КИМ разработаны с учетом того, что дисциплина включает следующие модули:
- Backend;
- Frontend;
- Интеграция и развертывание;
- Защита проекта.

---

## 2. Назначение КИМ

Настоящие контрольно-измерительные материалы предназначены для проверки:
- теоретических знаний по дисциплине;
- практических умений по разработке веб-приложения;
- навыков взаимодействия frontend, backend и модели машинного обучения;
- умений по подготовке и развертыванию проекта;
- способности обучающегося обосновывать принятые архитектурные и технологические решения;
- готовности обучающегося к защите выполненного проекта.

---

## 3. Проверяемые результаты обучения

По завершении дисциплины обучающийся должен уметь:

1. Проектировать и реализовывать REST API для работы с данными и моделью.
2. Разрабатывать пользовательский интерфейс на основе Vue для загрузки данных, запуска предсказаний и отображения результатов.
3. Интегрировать frontend, backend и модель в единое приложение.
4. Подготавливать проект к запуску и базовому развертыванию.
5. Обосновывать архитектурные и технологические решения, принятые в проекте.
6. Представлять результаты работы и отвечать на вопросы по реализованному решению.

---

## 4. Условия проведения

Дифференцированный зачет проводится в комбинированной форме, которая учитывает выполнение заданий из каждого модуля и защиту проекта. Выполнение дополнительных заданий студентом в рамках зачета остается на усмотрение преподавателя и может использоваться для дополнительной проверки знаний, в том числе в случае спорной оценки.

**Первый сценарий (нормальный):** для выставления оценки используются результаты промежуточной проверки знаний в рамках каждого модуля и защиты проекта.  
**Второй сценарий (расширенный):** используется, например, при возникновении спорных ситуаций с оцениванием, необходимости дополнительной проверки знаний по отдельным (или всем) разделам. В этом случае возможно повторное проведение тестов и заданий (возможно расширенных) на усмотрение преподавателя.

Допускается использование:
- учебных материалов дисциплины;
- официальной документации по используемым технологиям;
- справочных материалов, разрешенных преподавателем.

Использование генеративного искусственного интеллекта допускается только в случаях и пределах, установленных локальными нормативными актами образовательной организации.

---

## 5. Структура КИМ

### 5.1. Модуль 1. Backend

Модуль 1 включает выполнение заданий, направленных на проверку знаний и умений в области серверной разработки, проектирования и реализации REST API, а также обработки данных и запросов.

**Максимальный балл:** 50

---

### 5.2. Модуль 2. Frontend

Модуль 2 включает выполнение заданий, направленных на проверку знаний и умений в области разработки пользовательского интерфейса на основе Vue, взаимодействия с backend-сервисом и отображения результатов обработки данных.

**Максимальный балл:** 50

---

### 5.3. Модуль 3. Интеграция и развертывание

Модуль 3 включает выполнение заданий, направленных на проверку знаний и умений в области объединения frontend, backend и модели машинного обучения в единое решение, подготовки проекта к запуску, настройки окружения и базового развертывания.

**Максимальный балл:** 50

---

### 5.4. Модуль 4. Защита проекта

Модуль 4 включает защиту индивидуального проекта, в рамках которой обучающийся представляет выполненное решение, обосновывает принятые архитектурные и технологические решения, объясняет принципы работы приложения и отвечает на вопросы по проекту.

**Максимальный балл:** 100

**Проверяемые аспекты:**
- постановка задачи и обоснование решения;
- архитектура и организация приложения;
- реализация backend;
- реализация API для работы с данными;
- реализация frontend на основе Vue;
- развертывание и использование модели;
- промышленные практики разработки и эксплуатации;
- качество кода и тестируемость;
- защита проекта.

---

## 6. Порядок проведения

Дифференцированный зачет проводится поэтапно:

1. Выполнение заданий по модулю 1.
2. Выполнение заданий по модулю 2.
3. Выполнение заданий по модулю 3.
4. Защита проекта (модуль 4).

Результаты по каждому модулю фиксируются отдельно и используются для формирования итоговой оценки.

---

## 7. Критерии оценивания

### 7.1. Оценивание модулей 1–3

Каждый из модулей 1–3 оценивается отдельно по шкале:
- **43-50 баллов** — полный и корректный результат;
- **36–42 баллов** — в целом верное выполнение с несущественными недочетами;
- **28–35 баллов** — задание выполнено частично, имеются заметные ошибки;
- **0–27 баллов** — задание выполнено неудовлетворительно.

### 7.2. Оценивание модуля 4

Модуль 4 оценивается по следующей шкале:
- **86-100 баллов** — проект защищён полностью и без существенных замечаний;
- **71–85 баллов** — проект защищён в целом успешно, допущены отдельные недочёты;
- **56–70 баллов** — проект защищён с заметными ошибками или неполной реализацией отдельных требований;
- **0–55 баллов** — проект не соответствует требованиям или не может быть защищён в полном объёме.

### 7.3. Итоговая сумма баллов

Итоговая оценка формируется по сумме баллов, набранных за все модули:

- Модуль 1 — 50 баллов;
- Модуль 2 — 50 баллов;
- Модуль 3 — 50 баллов;
- Модуль 4 — 100 баллов.

**Итоговый максимум: 250 баллов.**

### 7.4. Перевод баллов в пятибалльную систему

| Сумма баллов | Оценка | Краткое описание |
|---|---|---|
| 225–250 | **5 (отлично)** | Обучающийся демонстрирует полное и уверенное владение материалом, успешно выполняет задания всех модулей и убедительно защищает проект. |
| 175–224 | **4 (хорошо)** | Обучающийся в целом правильно выполняет задания, допускает отдельные недочеты, но показывает достаточный уровень владения материалом и способен объяснить основные решения. |
| 125–174 | **3 (удовлетворительно)** | Обучающийся выполняет основные требования, но демонстрирует неполное понимание отдельных разделов, допускает существенные недочеты в реализации или защите проекта. |
| 0–124 | **2 (неудовлетворительно)** | Обучающийся не выполняет значительную часть требований, проект и/или отдельные модули содержат существенные ошибки либо не могут быть в полной мере защищены. |

---

## 8. Правила использования внешних ресурсов и ИИ

- Использование учебных материалов и официальной документации допускается в пределах, установленных преподавателем.
- Использование генеративного ИИ допускается только при наличии соответствующего разрешения.
- Запрещается использование ИИ для подмены самостоятельного выполнения заданий и защиты проекта, если это противоречит правилам дисциплины.
- При защите проекта обучающийся должен уметь объяснить представленный результат без опоры на внешние источники.

---

## 9. Порядок пересдачи

При неудовлетворительном результате обучающемуся может быть назначена пересдача в соответствии с локальными нормативными актами образовательной организации. Формат пересдачи сохраняет структуру дисциплины: задания по модулям и защита проекта.
