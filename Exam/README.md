# Экзамен / зачет

Промежуточная аттестация подтверждает освоение дисциплины в целом и не должна дублировать только один модуль.

# КОНТРОЛЬНО-ИЗМЕРИТЕЛЬНЫЕ МАТЕРИАЛЫ
## для проведения дифференцированного зачета по дисциплине

---

## 1. Общие положения

Контрольно-измерительные материалы (КИМ) предназначены для проведения дифференцированного зачета по дисциплине, направленной на освоение принципов разработки, интеграции и развертывания веб-приложения с моделью машинного обучения.

Дифференцированный зачет является формой промежуточной аттестации и служит для оценки степени освоения обучающимися содержания дисциплины в целом.

КИМ разработаны с учетом того, что дисциплина включает следующие модули:
- Backend;
- Frontend;
- Интеграция и развертывание;
- Защита проекта.

---

## 2. Назначение КИМ

Настоящие контрольно-измерительные материалы предназначены для проверки:
- теоретических знаний по дисциплине;
- практических умений по разработке веб-приложения;
- навыков взаимодействия frontend, backend и модели машинного обучения;
- умений по подготовке и развертыванию проекта;
- способности обучающегося обосновывать принятые архитектурные и технологические решения;
- готовности обучающегося к защите выполненного проекта.

---

## 3. Проверяемые результаты обучения

По завершении дисциплины обучающийся должен уметь:

1. Проектировать и реализовывать REST API для работы с данными и моделью.
2. Разрабатывать пользовательский интерфейс на основе Vue для загрузки данных, запуска предсказаний и отображения результатов.
3. Интегрировать frontend, backend и модель в единое приложение.
4. Подготавливать проект к запуску и базовому развертыванию.
5. Обосновывать архитектурные и технологические решения, принятые в проекте.
6. Представлять результаты работы и отвечать на вопросы по реализованному решению.

---

## 4. Условия проведения

Дифференцированный зачет проводится в комбинированной форме и включает выполнение дополнительных заданий и практических заданий по модулям дисциплины, а также защиту проекта.<br>Выполнение дополнительных заданий студентом остаётся на усмотрение преподавателя. И может использоваться для дополнительной проверки знаний, например, в случае спорной оценки. Для оценивания возможно использование результатов промежуточной проверки знаний в рамках каждого модуля.

Допускается использование:
- учебных материалов дисциплины;
- официальной документации по используемым технологиям;
- справочных материалов, разрешенных преподавателем.

Использование генеративного искусственного интеллекта допускается только в случаях и пределах, установленных локальными нормативными актами образовательной организации.

---

## 5. Структура КИМ

### 5.1. Модуль 1. Backend

**Цель:** проверить знание основ серверной разработки и умение реализовывать API для обработки запросов и данных.

**Дополнительные задания:**
1. Объясните, чем REST API отличается от произвольного набора HTTP-обработчиков.
2. Укажите, какие HTTP-методы следует использовать для чтения, создания, обновления и удаления ресурса, и поясните почему.
3. Сопоставьте коды ответов `200`, `201`, `400`, `404`, `422`, `500` с типичными ситуациями в API.
4. Опишите, какие проверки должны выполняться перед обработкой CSV-файла на сервере.
5. Предложите структуру JSON-ответа для случая успешного предсказания и для случая ошибки.
6. Объясните, почему входные данные нужно валидировать до обращения к модели.
7. Назовите причины, по которым сервер должен возвращать понятные сообщения об ошибке.
8. Проанализируйте пример псевдокода обработчика запроса и укажите в нём возможные ошибки проектирования.

**Практическое задание:**  
Реализовать один из сценариев:
- прием CSV-файла;
- проверка структуры данных;
- возвращение списка столбцов;
- выдача результата предсказания по одной записи.

**Что проверяется:**
- корректность маршрута;
- правильность формата запроса и ответа;
- обработка ошибок;
- использование подходящих HTTP-статусов;
- валидация входных данных.

---

### 5.2. Модуль 2. Frontend

**Цель:** проверить умение создавать пользовательский интерфейс на базе Vue для взаимодействия с backend-сервисом.

**Дополнительные задания:**
1. Опишите, какие основные элементы интерфейса должны присутствовать в приложении на Vue для загрузки данных и запуска предсказания.
2. Объясните, почему пользователь должен видеть состояния загрузки, успеха и ошибки при работе Vue-компонентов.
3. Предложите вариант структуры Vue-компонента или набора компонентов для формы загрузки CSV-файла.
4. Укажите, какие данные интерфейс на Vue должен передавать на сервер и какие получать в ответ.
5. Объясните, почему интерфейс на Vue не должен перегружаться лишними элементами.
6. Опишите, как можно отобразить результат работы модели в понятной для пользователя форме средствами Vue.
7. Проанализируйте макет интерфейса и отметьте его сильные и слабые стороны с точки зрения UX.
8. Объясните, какие ошибки на frontend чаще всего возникают при работе с API через Vue.

**Практическое задание:**  
Создать интерфейс на Vue, который позволяет:
- загрузить CSV-файл;
- отправить его на сервер;
- получить и отобразить результат обработки;
- показать сообщение об ошибке или успешном выполнении.

**Что проверяется:**
- наличие рабочей формы;
- корректное взаимодействие с API;
- отображение результата;
- удобство интерфейса;
- базовая логика обработки состояний;
- использование Vue-компонентов и реактивной логики.

---

### 5.3. Модуль 3. Интеграция и развертывание

**Цель:** проверить понимание принципов объединения frontend, backend и модели в единое решение, а также навыков подготовки проекта к запуску.

**Дополнительные задания:**
1. Объясните, почему frontend и backend должны согласовывать формат передаваемых данных.
2. Укажите, какие параметры приложения целесообразно выносить в конфигурацию.
3. Объясните назначение переменных окружения в проекте.
4. Сравните локальный запуск приложения и запуск в контейнере.
5. Назовите типичные проблемы при несовпадении версий библиотек.
6. Опишите, какие файлы должны входить в краткую инструкцию по запуску проекта.
7. Приведите пример структуры каталогов проектного решения и поясните назначение основных папок.
8. Объясните, какие признаки позволяют считать проект базово готовым к эксплуатации.

**Практическое задание:**  
Подготовить проект к запуску:
- описать структуру приложения;
- указать порядок запуска компонентов;
- настроить адрес API;
- подготовить краткую инструкцию по запуску.

**Что проверяется:**
- целостность решения;
- корректность взаимодействия компонентов;
- наличие базовой документации;
- воспроизводимость запуска;
- готовность проекта к использованию.

---

### 5.4. Модуль 4. Защита проекта

**Цель:** проверить умение обучающегося представить собственное решение, обосновать архитектурные и технологические решения, объяснить работу приложения и ответить на вопросы по проекту.

**Форма контроля:**  
Защита проекта по ранее разработанным критериям оценивания.

**Проверяемые аспекты:**
- постановка задачи и обоснование решения;
- архитектура и организация приложения;
- реализация backend;
- реализация API для работы с данными;
- реализация frontend на основе Vue;
- развертывание и использование модели;
- промышленные практики разработки и эксплуатации;
- качество кода и тестируемость;
- защита проекта.

---

## 6. Порядок проведения

Дифференцированный зачет проводится поэтапно:

1. Выполнение дополнительных заданий по модулю 1.
2. Выполнение практического задания по модулю 1.
3. Выполнение дополнительных заданий по модулю 2.
4. Выполнение практического задания по модулю 2.
5. Выполнение дополнительных заданий по модулю 3.
6. Выполнение практического задания по модулю 3.
7. Защита проекта по модулю 4.

Результаты по каждому модулю фиксируются отдельно и используются для формирования итоговой оценки.

---

## 7. Критерии оценивания

### 7.1. Оценивание модулей 1–3

Каждый из модулей 1–3 оценивается отдельно по шкале:
- **50 баллов** — полный и корректный ответ / выполнение задания;
- **40–49 баллов** — в целом верное выполнение с несущественными недочетами;
- **30–39 баллов** — задание выполнено частично, имеются заметные ошибки;
- **0–29 баллов** — задание выполнено неудовлетворительно.

### 7.2. Оценивание модуля 4

Модуль 4 оценивается по следующим уровням:
- **100 баллов** — проект защищён полностью и без существенных замечаний;
- **80–99 баллов** — проект защищён в целом успешно, допущены отдельные недочёты;
- **60–79 баллов** — проект защищён с заметными ошибками или неполной реализацией отдельных требований;
- **0–59 баллов** — проект не соответствует требованиям или не может быть защищён в полном объёме.

### 7.3. Итоговая сумма баллов

Итоговая оценка формируется по сумме баллов:
- Модуль 1 — 50 баллов;
- Модуль 2 — 50 баллов;
- Модуль 3 — 50 баллов;
- Модуль 4 — 100 баллов.

**Итоговый максимум: 250 баллов.**

---

## 8. Правила использования внешних ресурсов и ИИ

- Использование учебных материалов и официальной документации допускается в пределах, установленных преподавателем.
- Использование генеративного ИИ допускается только при наличии соответствующего разрешения.
- Запрещается использование ИИ для подмены самостоятельного выполнения заданий и защиты проекта, если это противоречит правилам дисциплины.
- При защите проекта обучающийся должен уметь объяснить представленный результат без опоры на внешние источники.

---

## 9. Порядок пересдачи

При неудовлетворительном результате обучающемуся может быть назначена пересдача в соответствии с локальными нормативными актами образовательной организации.  
Формат пересдачи сохраняет структуру дисциплины: дополнительные задания по модулям и защита проекта.

---

## 10. Заключительные положения

Настоящие контрольно-измерительные материалы могут быть использованы в составе фонда оценочных средств дисциплины. Подробное описание дисциплины, структуры модулей, планируемых результатов обучения и системы оценивания приводится в рабочей программе дисциплины.
