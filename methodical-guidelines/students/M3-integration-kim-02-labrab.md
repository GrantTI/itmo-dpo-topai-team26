# Методические рекомендации по выполнению лабораторной работы
## Модуль 3. Лабораторная работа «Развёртывание клиент-серверного приложения с моделями машинного обучения в контейнерной среде»

---

## 1. Вводная часть

**Тема:** контейнеризация сервисов, оркестрация Docker Compose, переменные окружения, базовая публикация через Nginx.

**Цель:** настроить и проверить воспроизводимый запуск рабочей версии **учебного** клиент-серверного приложения с моделями машинного обучения (модель — внутри backend модуля 1).

### Формат проведения

- **Тип работы:** лабораторная работа
- **Формат:** индивидуальное выполнение
- **Сдача:** файлы контейнеризации в репозитории; устная защита с демонстрацией контура; **письменный отчёт по работе**, представленный и защищённый на занятии

### Время выполнения

- **Общее время:** 6 академических часов
- **Распределение по этапам:**
  - Этап 1 (образы) — 1,5 часа
  - Этап 2 (Compose) — 1,5 часа
  - Этап 3 (переменные окружения) — 1 час
  - Этап 4 (Nginx и проверка с моделью) — 2 часа

### Необходимые ресурсы

| Инструмент | Версия | Назначение |
|---|---|---|
| Docker | 20.10+ | Контейнеры |
| Docker Compose | 2.0+ | Многосервисный контур |
| Git | 2.30+ | Контроль версий |
| IDE | актуальная | Редактирование конфигураций |

**Исходные материалы:** работоспособные серверная (модуль 1) и клиентская (модуль 2) части; рекомендуется результат лабораторной работы «Интеграция клиентской и серверной частей приложения с применением модели машинного обучения» (интегрированный клиент и сервер).

**Объект выполнения (по умолчанию):** учебный стенд = backend (FastAPI) + frontend (статика) + Nginx. Перенос на **собственный** проект — самостоятельная работа «Развёртывание итогового приложения дисциплины в контейнерной среде».

**Оборудование:** рекомендуется 8–16 ГБ ОЗУ, ≥10 ГБ свободного места.

## 2. Подготовка к выполнению

### Теоретический минимум

1. **Сеть Compose.** Контейнеры общаются по **имени сервиса** (`http://backend:8000`), а не по `localhost` хоста. `localhost` внутри контейнера — это сам контейнер [4; 7; 8].

2. **Единая точка входа.** Браузер открывает только опубликованный порт Nginx; Nginx проксирует статику UI и `/api` на backend [1; 6]. Модель остаётся **внутри** процесса backend (in-process, модуль 1) [2].

```
Браузер → Nginx (:8080) → frontend (статика) / backend (API) → model внутри backend
```

3. **Сборка frontend.** В сдаваемом контуре — статика (`npm run build`), не долгоживущий `npm run dev`. Образ frontend либо сам раздаёт статику (Nginx/`serve`), либо статика монтируется в edge-Nginx [3].

4. **Конфигурация через env.** Адреса, порты, режимы — из переменных окружения / `env_file`; в Git только `.env.example` без секретов [1].

### Настройка рабочего окружения

```

```

## 3. Алгоритм выполнения

### Общая стратегия

1. Соберите образы → опишите Compose → env → Nginx → проверка API модели.
2. После каждого этапа: `docker compose up --build` / `down`.
3. Не подменяйте объект работы собственным проектом (это самостоятельная работа «Развёртывание итогового приложения дисциплины в контейнерной среде»).

---

### Этап 1. Контейнерные образы

#### Backend (ориентир)

```dockerfile
FROM python:3.11-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY . .
CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000"]
```

Адаптируйте точку входа под структуру вашего модуля 1.

**Шаблон `backend/.dockerignore`:**

```text
__pycache__
*.pyc
.env
.venv
.git
.pytest_cache
*.md
```

#### Frontend (ориентир, multi-stage)

```dockerfile
FROM node:20-alpine AS build
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

FROM nginx:1.25-alpine
COPY --from=build /app/dist /usr/share/nginx/html
```

Путь к артефакту сборки (`dist` / `build`) согласуйте со стеком модуля 2. Многостадийная сборка frontend уже в этой лабораторной работе допустима и желательна; в лабораторной работе «Конфигурирование обратного прокси-сервера и многостадийная сборка контейнерных образов» её углубляют и/или применяют также к backend [9].

**Шаблон `frontend/.dockerignore`:**

```text
node_modules
dist
.env
.env.*
!.env.example
.git
```

Если сборке нужны переменные `VITE_*`, задайте их как `ARG`/`ENV` на стадии `build` или передайте через Compose `build.args` (без секретов в образе).
---

### Этап 2. Docker Compose

Пример минимального контура:

```yaml
services:
  backend:
    build: ./backend
    env_file: .env
    networks: [appnet]

  frontend:
    build: ./frontend
    networks: [appnet]

  nginx:
    image: nginx:1.25-alpine
    ports:
      - "8080:80"
    volumes:
      - ./nginx/default.conf:/etc/nginx/conf.d/default.conf:ro
    depends_on:
      - backend
      - frontend
    networks: [appnet]

networks:
  appnet:
```

Взаимодействие: `http://backend:8000` из Nginx, не `localhost:8000` хоста.

**Полезные команды:**

```bash
docker compose up --build -d
docker compose ps
docker compose logs -f backend
docker compose down
```

---

### Этап 3. Переменные окружения

`.env.example` (без секретов):

```env
# пример имён — адаптируйте под проект
BACKEND_HOST=backend
BACKEND_PORT=8000
```

Параметры, влияющие на CORS в контейнере (если ещё используются), должны учитывать origin через Nginx (часто `http://localhost:8080`).

---

### Этап 4. Nginx и проверка с моделью

Базовый `nginx/default.conf` (ориентир):

```nginx
server {
    listen 80;

    location / {
        proxy_pass http://frontend:80;
        proxy_set_header Host $host;
    }

    location /api/ {
        proxy_pass http://backend:8000/api/;
        proxy_set_header Host $host;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }
}
```

Согласуйте префиксы с реальными путями FastAPI и клиента. Для путей вида `/api/models` используйте `proxy_pass http://backend:8000/api/;` (сохранение префикса), а не `http://backend:8000/;`.

**Проверка после `docker compose up --build`:**

1. Открыть UI: `http://localhost:8080`
2. Smoke-тест модели через Nginx, например: `GET http://localhost:8080/api/models` (или эквивалент) — **недостаточно** одного health без моделей
3. Отличие от лабораторной работы «Интеграция клиентской и серверной частей приложения с применением модели машинного обучения»: здесь тот же класс проверки, но через Compose и единую точку входа
4. На защите: инференс внутри контейнера `backend` [2]

```bash
curl -s -o /dev/null -w "%{http_code}\n" http://localhost:8080/
curl -s http://localhost:8080/api/models
```

Углубление Nginx (кэш, балансировка) и multi-stage — лабораторная работа «Конфигурирование обратного прокси-сервера и многостадийная сборка контейнерных образов».

## 4. Оформление результатов

**Сдать:** Dockerfile, `docker-compose.yml`, Nginx-конфиг, `.env.example`, демонстрация запуска; **письменный отчёт** (см. требования ниже).

В README — команды:

```bash
cp .env.example .env
docker compose up --build -d
docker compose down
```

### Требования к отчёту

Отчёт должен содержать следующие пункты:

1. **Цель работы** — согласно цели в §1 настоящего МУ.
2. **Постановка задачи** — краткая постановка по §2–§3: состав контура, требования к воспроизводимому запуску и проверке через Nginx (без пересказа всей теории).
3. **Ход работы** — описание этапов выполнения, соответствующих постановке задачи и алгоритму §3; на каждом этапе — основные результаты. В разделе приводятся, в частности:
   - скриншоты сборки и запуска контура (`docker compose up --build` и т.п.);
   - скриншоты доступности UI через опубликованный порт Nginx;
   - скриншот или фрагмент ответа smoke-проверки метода API, связанного с моделью (не только health);
   - ключевые фрагменты конфигурации (Dockerfile, `docker-compose.yml`, Nginx, `.env.example`) либо ссылки на файлы в репозитории с пояснением назначения;
   - краткое описание состава сервисов, сети Compose и пути запроса браузер → Nginx → backend → применение модели.
4. **Выводы** — заключение о результатах: достижение цели работы, воспроизводимость запуска, место применения модели в контуре, ограничения решения; что сознательно отложено на лабораторную работу «Конфигурирование обратного прокси-сервера и многостадийная сборка контейнерных образов» и самостоятельную работу «Развёртывание итогового приложения дисциплины в контейнерной среде».

Максимум — **10 баллов**. Зачёт — не ниже **6**.

| Критерий | Баллы | Показатели |
|---|:---:|---|
| Сборка образов | 0–2 | Dockerfile; образы собираются |
| Оркестрация сервисов | 0–2 | Несколько сервисов; сеть; up/down |
| Конфигурация окружения | 0–2 | Env; `.env.example`; нет секретов в Git |
| Публикация и проверка контура | 0–2 | Nginx; UI; через Nginx — успешный ответ метода, связанного с моделью (smoke) |
| Обоснование решения | 0–2 | Состав контура; путь запроса; место инференса (backend) |
