## 1. Вводная часть

**Тема.** Организация взаимодействия клиентской и серверной частей приложения; способы размещения и вызова модели машинного обучения в клиент-серверной архитектуре.

**Цель работы.** Научиться сопоставлять инженерные варианты интеграции компонентов и обосновывать выбор применительно к учебному стенду или собственному проекту дисциплины.

- **Формат:** индивидуальная письменная работа
- **Сдача:** один файл Markdown или PDF; при необходимости — краткие устные уточнения на занятии

- **Общее время:** около 2 академических часов самостоятельной работы после лекций «Интеграция клиентской и серверной частей приложения» и «Артефакты моделей и паттерны интеграции инференса в клиент-серверную архитектуру»
- Указанное время не изменяет объём аудиторных занятий модуля

- Краткое описание учебного стенда дисциплины либо собственного проекта модулей 1–2 (назначение, стек, текущий способ вызова модели)
- Текстовый редактор

## 2. Подготовка к выполнению

### Теоретический минимум

1. **CORS и reverse-proxy — разные механизмы.**  
   Политика CORS разрешает браузеру выполнять кросс-доменные запросы к API при разных origin (типично: UI на `:5173`, API на `:8000`) [5]. Обратный прокси-сервер (Nginx) публикует UI и API под **одним** origin, поэтому для браузера кросс-доменной ситуации часто нет [6]. Для локальной разработки обычно достаточно CORS и/или Vite-proxy; для контейнерного контура предпочтителен единый origin через Nginx — в лабораторных работах «Развёртывание клиент-серверного приложения с моделями машинного обучения в контейнерной среде» и «Конфигурирование обратного прокси-сервера и многостадийная сборка контейнерных образов» [1].

2. **Контракт REST API** — согласованные метод, путь, схемы запроса/ответа и коды ошибок между клиентом (модуль 2) и сервером (модуль 1) [1; 3; 4]. Без контракта UI и backend «разъезжаются» даже при рабочем CORS.

3. **Три способа размещения инференса** (лекция «Артефакты моделей и паттерны интеграции инференса в клиент-серверную архитектуру»):
   - **in-process** — модель вызывается внутри процесса серверного приложения (как в модуле 1: FastAPI → `model.predict`);
   - **выделенный сервис** — отдельный процесс/контейнер только для предсказаний;
   - **внешний API** — облачный или сторонний HTTP-сервис.  
   Для учебного стенда дисциплины рабочий путь — **in-process** в backend модуля 1; остальные паттерны сравниваются в эссе, но в лабораторных работах «Интеграция клиентской и серверной частей приложения с применением модели машинного обучения», «Развёртывание клиент-серверного приложения с моделями машинного обучения в контейнерной среде» и «Конфигурирование обратного прокси-сервера и многостадийная сборка контейнерных образов» отдельно не реализуются [2].

4. **Связь с жизненным циклом ML-системы.** Размещение инференса влияет на изоляцию сбоев, задержки, версионирование модели и сложность развёртывания — это предмет сравнения в задании 3 [2].

### Настройка рабочего окружения

Специальная программная среда не требуется.

### Анализ задания

Работа состоит из **четырёх обязательных заданий** (см. §3). Объём связного текста — **400–700 слов** (таблицы в лимит не входят). Сценарии computer vision и RAG допускаются только как краткий обзорный абзац (не более одного); компетенции по большим языковым моделям и глубокому обучению в работе не заявляются.

## 3. Алгоритм выполнения

### Общая стратегия

1. Сначала заполните таблицы, затем напишите выводы и выбор — так проще уложиться в лимит слов.
2. В таблицах сравнивайте по **критериям**, а не пересказывайте определения из лекции.
3. Выбор в задании 4 должен опираться на строки таблиц.

---

### Задание 1. Контекст приложения

В одном абзаце (**100–120 слов**) укажите:

- назначение приложения;
- серверную и клиентскую части стека;
- текущий способ вызова модели (например, внутрипроцессный вызов через API модуля 1).

---

### Задание 2. Сравнение вариантов интеграции клиента и сервера

Выберите **не менее двух** вариантов из перечня:

1. Политика CORS на стороне сервера.
2. Публикация через обратный прокси-сервер (единый origin для клиента и API).
3. Прямое обращение клиента к API без прокси при согласованных адресах.

Составьте таблицу **не менее чем по четырём критериям** (например: сложность настройки; безопасность в учебной среде; удобство локальной разработки; пригодность к контейнерному развёртыванию; прозрачность маршрутизации). После таблицы дайте краткие выводы по критериям.

### Задание 4. Обоснованный выбор и ограничения

Объём раздела — **150–250 слов**.

1. Явно выберите вариант интеграции клиента и сервера (задание 2) и способ размещения инференса (задание 3) для лабораторных работ модуля 3 и/или собственного проекта. Для учебного стенда дисциплины ожидаемый рабочий способ — **внутрипроцессный инференс в backend модуля 1**; выделенный и внешний API сравниваются в таблице, но в лабораторных работах «Интеграция клиентской и серверной частей приложения с применением модели машинного обучения», «Развёртывание клиент-серверного приложения с моделями машинного обучения в контейнерной среде» и «Конфигурирование обратного прокси-сервера и многостадийная сборка контейнерных образов» отдельно не реализуются, если иное не согласовано для своего проекта [2].
2. Обоснуйте выбор ссылкой на критерии таблиц (не формулировкой «так удобнее»).
3. Укажите ограничения решения и то, что остаётся вне объёма модуля 3 (например, оркестрация контейнеров средствами Kubernetes, полноценный CI/CD).
4. Будьте готовы устно связать выбор с лабораторной работой «Интеграция клиентской и серверной частей приложения с применением модели машинного обучения» и последующим развёртыванием в лабораторных работах «Развёртывание клиент-серверного приложения с моделями машинного обучения в контейнерной среде» и «Конфигурирование обратного прокси-сервера и многостадийная сборка контейнерных образов».
