# Методические рекомендации по выполнению лабораторной работы
## Модуль 3. Лабораторная работа «Интеграция клиентской и серверной частей приложения с применением модели машинного обучения»

---

## 1. Вводная часть

**Тема:** приёмка связки модулей 1 и 2 перед Docker: контракт API, доступ браузера к API, сквозной сценарий **из UI** с моделью.

**Цель:** показать, что клиентская часть обращается к серверной и что из интерфейса доступна функциональность модели (API модуля 1).

### Формат проведения

- **Тип работы:** лабораторная работа
- **Формат:** индивидуальное выполнение
- **Сдача:** код в репозитории; **обязательная демонстрация из UI**; примеры окружения без секретов; **письменный отчёт по работе**, представленный и защищённый на занятии

### Время выполнения

- **Общее время:** 2 академических часа
- **Распределение по этапам:**
  - Этап 1 (контракт и базовый URL) — 0,4 часа
  - Этап 2 (подтверждение CORS/proxy) — 0,4 часа
  - Этап 3 (сквозной сценарий из UI с моделью) — 1,2 часа

### Необходимые ресурсы

| Инструмент | Назначение |
|---|---|
| Python 3.x, FastAPI (модуль 1) | Серверная часть с API моделей |
| Стек клиентской части модуля 2 (Vue/Vite и т.п.) | UI — обязателен для сдачи |
| Git | Контроль версий |

**Важно.** CORS и Vite-proxy часто уже настроены в модуле 2 [3; 4]. Если запросы из UI к API уже проходят — на защите достаточно это подтвердить. Новый акцент работы — **сценарий с моделью из интерфейса**, а не повторная настройка CORS «с нуля». Рекомендуется опереться на выводы сравнительно-аналитического эссе «Интеграция клиентской и серверной частей приложения и способы подключения моделей машинного обучения» (паттерн in-process для учебного стенда) [2].

## 2. Подготовка к выполнению

### Теоретический минимум

1. **Контракт API модуля 1** — пути вроде `/api/models`, `/api/train`, `/api/predict` (уточните по своему OpenAPI). Клиент модуля 2 должен вызывать **те же** пути и понимать схемы ответов.

2. **Два способа обойти разные origin в разработке:**
   - **CORS на FastAPI** — сервер явно разрешает origin фронтенда (`Access-Control-Allow-Origin` и связанные заголовки) [5; 4]. Удобно, когда UI ходит напрямую на `http://127.0.0.1:8000`.
   - **Dev-proxy (Vite)** — браузер видит только origin UI (`:5173`), а запросы к `/api` проксируются на backend [7]. Кросс-доменного запроса с точки зрения браузера нет.  
   В лабораторных работах «Развёртывание клиент-серверного приложения с моделями машинного обучения в контейнерной среде» и «Конфигурирование обратного прокси-сервера и многостадийная сборка контейнерных образов» вместо Vite-proxy обычно ставят **Nginx** с единым origin [1; 6].

3. **In-process инференс** (лекция «Артефакты моделей и паттерны интеграции инференса в клиент-серверную архитектуру», практика модуля 1): UI → FastAPI → `model.predict` **в том же процессе**. Обоснование паттернов — [2].

```
Браузер (UI модуля 2) → HTTP → Backend (FastAPI, модуль 1) → model внутри процесса
```

4. **Переменные окружения** — единая точка настройки базового URL API на клиенте (`VITE_API_BASE_URL` или относительный `/api`). Не дублируйте «магические» URL в каждом компоненте.

### Настройка рабочего окружения

1. Запустите backend модуля 1.
2. Запустите frontend модуля 2.
3. Убедитесь, что есть экран/действие UI, которое вызывает API моделей (список, обучение или прогноз).

### Анализ задания

| Этап | Результат |
|---|---|
| 1 | Единый базовый URL; `.env.example`; список методов |
| 2 | Запросы из UI к API работают (CORS и/или proxy) |
| 3 | **Из UI** показан вызов функциональности модели |

Сдача **только** через Swagger/`curl` **не засчитывается** как замена UI.

## 3. Алгоритм выполнения

### Этап 1. Контракт и базовый адрес API

1. Точка конфигурации на клиенте (`VITE_API_BASE_URL` или относительный `/api` при proxy).
2. `.env.example` без секретов.
3. Для защиты — не менее двух методов, один из которых связан с моделями.

**Шаблон `.env.example` (frontend):**

```env
# при прямых запросах на backend:
VITE_API_BASE_URL=http://127.0.0.1:8000

# при Vite-proxy обычно достаточно относительных путей /api/...
# VITE_API_BASE_URL=
```

**Шаблон клиента API (Vue / axios):**

```js
// src/api/client.js
import axios from "axios";

const baseURL = import.meta.env.VITE_API_BASE_URL || "";

export const api = axios.create({
  baseURL,
  timeout: 15000,
});

export async function listModels() {
  const { data } = await api.get("/api/models");
  return data;
}
```

Адаптируйте пути под OpenAPI вашего модуля 1.

---

### Этап 2. CORS и/или прокси

Если уже сделано в M2 и UI ходит в API — переходите к этапу 3, на защите кратко поясните схему.

Иначе донастройте один из вариантов.

**Вариант A — CORS на FastAPI:**

```python
# ориентир; подключите к своему app
import os
from fastapi.middleware.cors import CORSMiddleware

origins = [
    o.strip()
    for o in os.getenv("CORS_ORIGINS", "http://127.0.0.1:5173").split(",")
    if o.strip()
]

app.add_middleware(
    CORSMiddleware,
    allow_origins=origins,
    allow_credentials=True,
    allow_methods=["*"],
    allow_headers=["*"],
)
```

```env
# backend/.env.example
CORS_ORIGINS=http://127.0.0.1:5173,http://localhost:5173
```

**Вариант B — Vite `server.proxy`:**

```js
// vite.config.js
import { defineConfig } from "vite";
import vue from "@vitejs/plugin-vue";

export default defineConfig({
  plugins: [vue()],
  server: {
    proxy: {
      "/api": {
        target: "http://127.0.0.1:8000",
        changeOrigin: true,
      },
    },
  },
});
```

При proxy на клиенте вызывайте относительные пути (`/api/models`), без полного URL на `:8000`.

Поясните на защите: в лабораторной работе «Развёртывание клиент-серверного приложения с моделями машинного обучения в контейнерной среде» обычно будет единый origin через Nginx, а не Vite-dev [1; 6].

### Этап 3. Сквозной сценарий из UI

1. В интерфейсе выполните действие, которое обращается к API модели (минимум — отображение списка моделей; лучше — train/predict по готовности стенда).
2. Остановите backend: UI должен показать понятную ошибку, а не «молча сломаться» без диагностики.
3. На защите свяжите результат с паттерном in-process (и со сравнительно-аналитическим эссе «Интеграция клиентской и серверной частей приложения и способы подключения моделей машинного обучения», если оно уже сдано) [2].

**Шаблон обработки ошибки в UI (ориентир):**

```js
try {
  const models = await listModels();
  // обновить состояние компонента
} catch (err) {
  const message =
    err.response?.data?.detail ||
    err.message ||
    "Сервер недоступен или вернул ошибку";
  // показать message пользователю (toast / alert / блок ошибки)
}
```

**Дополнительная проверка контракта (не заменяет UI):**

```bash
curl -s http://127.0.0.1:8000/api/models
```

## 4. Оформление результатов

**Сдать:** код frontend+backend; `.env.example`; демонстрация **из UI**; **письменный отчёт** (см. требования ниже).

```markdown
## Локальная интеграция (данная лабораторная работа)
1. Backend: ...
2. Frontend: ...
3. CORS / proxy: ...
4. Демо из UI: GET /api/models (или train/predict)
```

### Требования к отчёту

Отчёт должен содержать следующие пункты:

1. **Цель работы** — согласно цели в §1 настоящего МУ.
2. **Постановка задачи** — краткая постановка по §2–§3: что должно быть достигнуто в рамках данной лабораторной работы (без пересказа всей теории).
3. **Ход работы** — описание этапов выполнения, соответствующих постановке задачи и алгоритму §3; на каждом этапе — основные результаты. В разделе приводятся, в частности:
   - скриншоты пользовательского интерфейса со сквозным сценарием, связанным с моделью;
   - фрагменты конфигурации доступа браузера к API (CORS и/или dev-proxy) и задания базового URL;
   - скриншот или описание реакции UI при недоступности серверной части;
   - перечень продемонстрированных методов API (в том числе связанных с моделями).
4. **Выводы** — заключение о результатах: достижение цели работы, что подтверждено демонстрацией из UI, ограничения решения и отличие локальной интеграции от последующего контейнерного контура (лабораторная работа «Развёртывание клиент-серверного приложения с моделями машинного обучения в контейнерной среде»).
