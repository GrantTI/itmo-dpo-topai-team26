---

## 1. Вводная часть

**Тема:** уточнение маршрутизации Nginx; элементы устойчивости и эффективности; многостадийная сборка образов.

**Цель:** кастомизировать контур лабораторной работы «Развёртывание клиент-серверного приложения с моделями машинного обучения в контейнерной среде» — разделить статику и API, реализовать два элемента устойчивости/эффективности по правилу ниже, оптимизировать сборку не менее одного образа.

### Формат проведения

- **Тип работы:** лабораторная работа
- **Формат:** индивидуальное выполнение
- **Сдача:** обновлённые Nginx и Dockerfile; устная защита с демонстрацией; **письменный отчёт по работе**, представленный и защищённый на занятии

### Время выполнения

- **Общее время:** 6 академических часов
- Этап 1 (маршрутизация) — 1,5 ч; этап 2 (два элемента) — 2 ч; этап 3 (multi-stage) — 1,5 ч; этап 4 (проверка) — 1 ч

### Необходимые ресурсы

Рабочий контур лабораторной работы «Развёртывание клиент-серверного приложения с моделями машинного обучения в контейнерной среде»; Docker и Docker Compose; IDE. Рекомендуется 8–16 ГБ ОЗУ.

## 2. Подготовка к выполнению

### Теоретический минимум

1. **Маршрутизация.** Статика UI и API — разные блоки `location`. Проксирование — на **имя сервиса** Compose [3; 4]. Ошибка с `proxy_pass` и завершающим `/` может срезать префикс `/api/` и сломать пути модуля 1 (`/api/models` → `/models`) [4].

2. **Таймауты прокси** (зафиксированные значения для работы) [4]:

```nginx
proxy_connect_timeout 5s;
proxy_send_timeout 30s;
proxy_read_timeout 30s;
```

3. **Устойчивость и эффективность (задание 2).** Нужны **ровно два** элемента по правилу ниже. Балансировка (`upstream`) имеет смысл, когда есть ≥2 экземпляра backend; кэш — для безопасных GET/статики; `limit_req` — защита API от шквала запросов [1; 4]. Это кастомизация контура под LC-5.1, а не замена ML-логики backend [2].

4. **Multi-stage.** Стадия сборки содержит компиляторы/`npm`/`pip` build-зависимости; стадия runtime — только то, что нужно для запуска. Меньше размер образа и поверхность атаки [3; 5].

### Правило выбора элементов задания 2

Реализовать **ровно два** пункта:

1. **Обязательно** один из позиций 1–2:
   - **(1)** балансировка / резервирование: `upstream` и не менее двух экземпляров backend;
   - **(2)** кэширование статики и/или безопасных ответов GET.
2. **Второй** пункт — любой из 1–3, включая **(3)** ограничение интенсивности запросов (`limit_req` / `limit_conn`) или отдельный `location` проверки состояния (health).

До начала работы запишите выбранную пару и способ демонстрации эффекта.

### Настройка

```bash
docker compose up --build -d
docker compose down
```

Сохраните копии `default.conf` и Dockerfile до правок.

## 3. Алгоритм выполнения

### Этап 1. Маршрутизация

Пример базы:

```nginx
server {
    listen 80;

    # Важно: URI после proxy_pass должен сохранять префикс /api/,
    # если пути FastAPI вида /api/models (как в модуле 1).
    location /api/ {
        proxy_pass http://backend:8000/api/;
        proxy_connect_timeout 5s;
        proxy_send_timeout 30s;
        proxy_read_timeout 30s;
    }

    location / {
        proxy_pass http://frontend:80;
        proxy_set_header Host $host;
    }
}
```

Если в лабораторной работе «Развёртывание клиент-серверного приложения с моделями машинного обучения в контейнерной среде» статика отдаётся самим edge-Nginx (`root` / volume), оставьте тот вариант; иначе проксируйте на сервис `frontend`, как в примере выше.  
Не используйте `proxy_pass http://backend:8000/;` для `location /api/`, если это превращает `/api/models` в `/models` на backend.

**Контроль:** статика и API разведены; прокси на сервис Compose; таймауты заданы.

---

### Этап 2. Два элемента устойчивости или эффективности

#### Вариант A. Upstream (балансировка)

В Compose — два сервиса backend (например, `backend1`, `backend2`) с одной сборкой.

```nginx
upstream backend_upstream {
    least_conn;
    server backend1:8000;
    server backend2:8000;
}

location /api/ {
    proxy_pass http://backend_upstream/api/;
    proxy_connect_timeout 5s;
    proxy_send_timeout 30s;
    proxy_read_timeout 30s;
}
```

**Демо:** серия запросов к `/api/...`; в логах/метриках видно распределение (или пояснение схемы на защите).

#### Вариант B. Кэширование

```nginx
proxy_cache_path /var/cache/nginx levels=1:2 keys_zone=api_cache:10m max_size=64m inactive=10m use_temp_path=off;

server {
    listen 80;

    location /api/health {
        proxy_pass http://backend:8000/health;
        proxy_cache api_cache;
        proxy_cache_valid 200 30s;
        proxy_cache_methods GET HEAD;
        add_header X-Cache-Status $upstream_cache_status;
    }

    location ~* \.(js|css|png|jpg|jpeg|gif|ico|svg|woff2?)$ {
        root /usr/share/nginx/html;
        expires 1h;
        add_header Cache-Control "public";
    }
}
```

Путь health замените на существующий безопасный GET вашего API. Для тома кэша в Compose при необходимости добавьте volume к Nginx.

**Демо:** два подряд GET; заголовок `X-Cache-Status` (MISS затем HIT) или эквивалент.

#### Вариант C. Ограничение запросов

```nginx
limit_req_zone $binary_remote_addr zone=api_limit:10m rate=5r/s;

server {
    location /api/ {
        limit_req zone=api_limit burst=10 nodelay;
        proxy_pass http://backend:8000/api/;
    }
}
```

**Демо:** серия быстрых запросов → ответы `503` (или иной код ограничения) при превышении.

#### Вариант D. Health-location

```nginx
location = /healthz {
    proxy_pass http://backend:8000/health;
    access_log off;
}
```

**Демо:** `curl http://localhost:8080/healthz`.

Реализуйте выбранную пару по правилу; оба эффекта покажите на защите.

### Этап 3. Многостадийная сборка

Не менее двух стадий; итоговый образ без инструментов сборки.

```dockerfile
FROM node:20-alpine AS build
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

FROM nginx:1.25-alpine
COPY --from=build /app/dist /usr/share/nginx/html
```

Для Python-backend допустима схема: стадия с установкой зависимостей → slim-образ только с runtime.

```dockerfile
# backend/Dockerfile (ориентир multi-stage)
FROM python:3.11-slim AS deps
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir --prefix=/install -r requirements.txt

FROM python:3.11-slim
WORKDIR /app
COPY --from=deps /install /usr/local
COPY . .
CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000"]
```

```bash
docker build -t app-frontend:multistage ./frontend
docker build -t app-backend:multistage ./backend
docker image ls
```

Подготовьте сравнение с одностадийным вариантом (размер образа и/или состав слоёв). Обновите Compose и проверьте UI и smoke API модели (например, `GET /api/models`).

## 5. Критерии оценки

Максимум — **10 баллов**. Зачёт — не ниже **6**.

| Критерий | Баллы | Показатели |
|---|:---:|---|
| Маршрутизация | 0–3 | Статика и API разделены; прокси на Compose; таймауты заданы |
| Два элемента устойчивости/эффективности | 0–3 | Правило выбора соблюдено; эффекты показаны |
| Многостадийная сборка | 0–2 | ≥2 стадий; образ рабочий; контур запускается |
| Обоснование | 0–2 | Директивы, стадии, путь запроса и smoke API модели через Nginx |
