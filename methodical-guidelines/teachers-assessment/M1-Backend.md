## 1. Общие положения

Настоящие методические указания предназначены для преподавателей, осуществляющих оценивание результатов освоения модуля «Backend» в рамках дисциплины «Технологии развертывания моделей машинного обучения в клиент-серверных приложениях».  
Модуль включает два вида аттестационных мероприятий:
+ самостоятельную аудиторную работу (КИМ-1.1)
+ лабораторную работу (КИМ-1.2).

Критерии оценивания каждого мероприятия зафиксированы в соответствующих учебно-методических материалах. В настоящих указаниях приведена сводная таблица вклада каждой части в итоговый балл модуля.

## 2. Оценивание самостоятельной аудиторной работы (КИМ-1.1)

Самостоятельная аудиторная работа выполняется студентами индивидуально в аудитории под наблюдением преподавателя. Работа включает шесть заданий, каждое из которых оценивается в соответствии с зафиксированными в КИМ критериями. Максимальный общий балл — 20.

### 2.1. Задание №1 «Найди 5 ошибок в C4-диаграмме»

Оценивание производится по трём критериям. Максимальный балл — 3.

| Критерий | Баллы |
|----------|-------|
| Найдено ≥5 ошибок в C4-диаграмме (за каждую верную ошибку — 0.2 балла) | 1 |
| Исправленная схема не содержит новых ошибок (Nginx не ходит в БД, Vue.js не представлен как контейнер) | 1 |
| Корректно подписана стрелка: указаны протокол, порт, направление запроса | 1 |
| **ИТОГО** | **3** |

**Примечание для преподавателя:** Ошибки засчитываются только при наличии обоснования. При исправлении схемы студент должен показать понимание роли каждого компонента в архитектуре.

### 2.7. Процедура оценивания самостоятельной работы

1. **Формат проведения:** Аудиторная работа, 45–60 минут.
2. **Материалы:** Студентам выдаются задания в печатном виде с диаграммами и текстовыми описаниями.
3. **Регламент:**
   - 5 минут — организационные моменты и инструктаж
   - 40 минут — выполнение заданий
   - 5 минут — сдача работ
4. **Проверка:**
   - Каждое задание проверяется отдельно по утверждённым критериям
   - Баллы суммируются для получения итогового балла
   - Работы проверяются в течение 3–5 рабочих дней

### 3.7. Процедура оценивания лабораторной работы

1. **Формат сдачи:**
   - Git-репозиторий с исходным кодом
   - README.md с инструкцией по запуску и примерами запросов
   - Скриншоты Swagger UI (`/docs`)
   - Отчёт с ответами на контрольные вопросы

2. **Порядок проверки:**
   - Преподаватель клонирует репозиторий студента
   - Запускает приложение (локально или через Docker)
   - Проверяет все эндпоинты через Swagger UI или Postman
   - Проверяет качество кода и соответствие архитектуре
   - Проверяет документацию и логирование

3. **Сроки:**
   - Работа сдаётся в течение 1 недели после выполнения
   - Проверка — до 5 рабочих дней
   - При необходимости — защита работы с демонстрацией кода и ответами на вопросы

4. **Защита лабораторной работы (по требованию преподавателя):**
   - Студент демонстрирует работающее приложение
   - Студент объясняет ключевые архитектурные решения
   - Студент отвечает на уточняющие вопросы по коду

## 5. Рекомендации по проверке качества кода

Для более объективной оценки лабораторной работы рекомендуется учитывать следующие критерии качества кода:

| Критерий | Описание | Влияние на оценку |
|----------|----------|-------------------|
| **Читаемость кода** | Осмысленные имена переменных и функций, структурированность | ±1 балл |
| **Типизация** | Использование аннотаций типов в Python | ±0.5 балла |
| **DRY** | Отсутствие дублирования кода | ±0.5 балла |
| **SOLID** | Соблюдение принципов SOLID в архитектуре | ±1 балл |
| **Обработка ошибок** | Все возможные ошибки обработаны | ±1 балл |
| **Безопасность** | Валидация входных данных, отсутствие хардкода | ±1 балл |
| **Тесты** | Наличие юнит-тестов (опционально) | +1 балл |

**Примечание:** Указанные критерии могут использоваться для корректировки оценки в пределах ±1 балла за этап. Максимальная суммарная корректировка не должна превышать ±3 балла за всю лабораторную работу.

## 6. Типичные ошибки студентов и рекомендации

### 6.1. По самостоятельной работе КИМ-1.1

| Типичная ошибка | Рекомендация по проверке | Снижение балла |
|-----------------|--------------------------|----------------|
| Путаница между уровнями C4 (Container vs Component) | Проверить, что студент не использует термины Component на уровне Container | -0.5 за задание |
| Nginx напрямую обращается к БД | Напомнить, что Nginx — только прокси, а не источник данных | -0.5 за задание |
| Отсутствие легенды на диаграмме | В C4-диаграмме легенда обязательна | -0.5 за задание |
| Неправильное направление стрелок | Направление = кто инициирует запрос | -0.5 за задание |
| Отсутствие версий технологий | У каждого контейнера должна быть версия | -0.5 за задание |
| Смешивание потоков данных и конфигурации | Использовать разные типы линий | -0.5 за задание |

### 6.2. По лабораторной работе КИМ-1.2

| Типичная ошибка | Рекомендация по проверке | Снижение балла |
|-----------------|--------------------------|----------------|
| Вся логика в main.py (отсутствие слоёв) | Проверить разделение на routes/services/schemas | -1 за этап 1 |
| Отсутствие CORS | Проверить настройку CORS middleware | -0.5 за этап 1 |
| Нет глобального состояния | Проверить наличие app_state | -1 за этап 1 |
| Хардкод переменных окружения | Должны быть через Pydantic Settings | -1 за этап 1 |
| Не все модели реализованы | Проверить наличие 5 моделей из ТЗ | -0.5 за каждую |
| Метрики считаются неверно | Проверить формулы метрик | -0.5 за этап 3 |
| Нет разделения на train/test | Проверить hold-out валидацию | -1 за этап 4 |
| Графики не генерируются | Проверить наличие файлов в exports/ | -1 за этап 4 |
| Нет проверки структуры данных при прогнозе | Проверить валидацию на этапе predict | -0.5 за этап 5 |
| Отсутствие глобального exception handler | Проверить обработку исключений | -1 за этап 6 |
| Swagger пустой или неинформативный | Проверить теги, докстринги, примеры | -1 за этап 6 |
