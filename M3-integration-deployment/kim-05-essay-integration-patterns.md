# Сравнительно-аналитическое эссе
## Тема: Интеграция клиентской и серверной частей приложения и паттерны подключения моделей машинного обучения

**Цель работы:** Сформировать умение анализировать и сопоставлять варианты организации HTTP-взаимодействия клиента и сервера, а также паттерны интеграции артефактов моделей машинного обучения в клиент-серверную архитектуру; обосновать выбор решений применительно к учебному или собственному проекту дисциплины.

**Продолжительность:** 2 академических часа самостоятельной работы (выполняется после занятий 3.1–3.2; не изменяет объём аудиторных занятий модуля).

**Связь с занятием:** 3.1–3.2 (подготовка к лабораторным работам 3.3–3.5)

**Проверяемые результаты обучения**

| Компетенция | Индикатор | Уровень | Дескриптор в контексте задания |
|---|---|---|---|
| PL-1. Способен применять язык программирования Python для решения задач в области ИИ | PL-1.1 | С | Сопоставляет варианты согласования контракта REST API и организации обращения к модели; формулирует обоснованный выбор применительно к серверному приложению на FastAPI |
| LC-5. Способен применять и (или) проектировать инструменты и инженерные практики промышленной разработки, развёртывания, эксплуатации и мониторинга систем ИИ | LC-5.1 | С | Анализирует инженерные варианты интеграции клиента и сервера (политика CORS, обратный прокси) и паттерны размещения инференса; обосновывает выбор для рабочей версии учебного приложения |

**Необходимые условия:** освоение содержания занятий 3.1 и 3.2; наличие описания учебного стенда дисциплины либо собственного проекта модулей 1–2 (достаточно краткого контекста: стек, наличие API моделей, наличие клиентской части).

**Форма контроля:** текущая (письменная работа с проверкой по рубрике; при необходимости — уточняющие вопросы преподавателя).

---

## Задание 1. Контекст приложения

**Описание.** Кратко зафиксировать объект сравнения: учебный стенд дисциплины или собственный проект.

**Требования:**

1. Указаны назначение приложения, стек (серверная и клиентская части) и способ применения модели на текущий момент (например, внутрипроцессный вызов через API модуля 1).
2. Объём раздела — один абзац (не более 100–120 слов).

---

## Задание 2. Сравнение вариантов интеграции клиента и сервера

**Описание.** Сопоставить не менее двух подходов к организации взаимодействия клиентской и серверной частей (содержание занятия 3.1).

**Варианты для сравнения (выбрать не менее двух):**

1. Разрешение кросс-доменных запросов на стороне сервера (политика CORS).
2. Публикация через обратный прокси-сервер (единый origin для клиента и API).
3. Прямое обращение клиента к API без прокси при согласованных адресах и политике доступа (если рассматривается в материалах занятия).

**Требования:**

1. Приведена таблица сравнения не менее чем по четырём критериям (например: сложность настройки; безопасность в учебной среде; удобство локальной разработки; пригодность к последующему контейнерному развёртыванию; прозрачность маршрутизации).
2. После таблицы дано краткое текстовое пояснение (не пересказ лекции, а выводы по выбранным критериям).
3. Не требуется приводить конфигурационные файлы; достаточно корректного инженерного рассуждения.

---

## Задание 3. Сравнение паттернов интеграции инференса

**Описание.** Сопоставить паттерны размещения инференса модели в клиент-серверной архитектуре (содержание занятия 3.2).

**Варианты для сравнения (обязательно все три):**

1. Внутрипроцессный инференс в серверном приложении (in-process).
2. Выделенный сервис инференса.
3. Внешний API инференса (облачный или сторонний HTTP-сервис).

**Требования:**

1. Приведена таблица сравнения не менее чем по четырём критериям (например: изоляция сбоев; сложность развёртывания; управление версией модели; задержки и сетевые риски; пригодность для классической табличной модели учебного проекта).
2. Сценарии computer vision и RAG допускаются только как краткий обзорный комментарий (не более одного абзаца); отдельные компетенции по LLM и глубокому обучению в работе не заявляются.
3. После таблицы сформулированы выводы, связанные с контекстом из задания 1.

---

## Задание 4. Обоснованный выбор и ограничения

**Описание.** Зафиксировать решения, которые обучающийся принимает для дальнейшего выполнения модуля 3.

**Требования:**

1. Явно выбран вариант интеграции клиента и сервера (задание 2) и паттерн инференса (задание 3) для ближайших лабораторных работ и/или собственного проекта. Для учебного стенда дисциплины ожидаемым рабочим паттерном является внутрипроцессный инференс в серверной части модуля 1; иные паттерны допускаются как обоснованная альтернатива для собственного проекта.
2. Выбор обоснован ссылкой на критерии таблиц, а не только на формулировку «так удобнее».
3. Указаны ограничения выбранного решения и то, что сознательно остаётся вне объёма модуля 3 (например, оркестрация контейнеров, полноценный CI/CD).
4. Объём раздела — 150–250 слов.

---

## Объём и оформление работы

1. Общий объём основного текста: **400–700 слов** (таблицы в лимит слов не входят; учитывается связный текст).
2. Структура обязательна: контекст → сравнение (3.1) → сравнение (3.2) → выбор и ограничения.
3. Язык — академический русский; англоязычные термины (CORS, reverse-proxy, in-process, inference) допускаются при первом пояснении.
4. Формат файла: Markdown или PDF; имя файла содержит фамилию обучающегося.

---

## Порядок выполнения и сдачи

1. Выполнить задания 1–4 после освоения занятий 3.1 и 3.2.
2. Сдать файл в срок, установленный преподавателем (рекомендуется до начала или в начале лабораторной работы 3.3).
3. При проверке быть готовым устно пояснить строки таблиц и связь выбора с лабораторной работой 3.3 (интеграция клиента и сервера) и последующим развёртыванием.

**Формат сдачи:** один файл эссе; при необходимости — краткие уточнения на занятии.
