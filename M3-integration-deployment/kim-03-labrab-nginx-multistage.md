# Лабораторная работа
## Тема: Конфигурирование reverse-proxy и оптимизация контейнерной сборки

**Цель работы:** Сформировать умение кастомизировать production-контур клиент-серверного ML-приложения: уточнить маршрутизацию в Nginx, применить элементы балансировки и кэширования, использовать multi-stage сборку образов.

**Продолжительность:** 6 академических часов

**Связь с занятием:** 3.5

**Проверяемые результаты обучения**

| Компетенция | Индикатор | Уровень | Дескриптор в контексте задания |
|---|---|---|---|
| LC-5. Способен применять и (или) проектировать инструменты и инженерные практики промышленной разработки, развёртывания, эксплуатации и мониторинга систем ИИ | LC-5.1 | С | Кастомизирует reverse-proxy и параметры контейнерной сборки для рабочей версии ML-приложения; настраивает маршрутизацию и элементы устойчивости/производительности публикации сервисов |

**Необходимые условия:** выполненная лабораторная работа занятия 3.4 (базовый Compose-контур с Nginx); Docker и Docker Compose; текстовый редактор.

**Форма контроля:** текущая (защита лабораторной работы).

---

## Задание 1. Уточнение маршрутизации в Nginx

**Описание.** Доработайте конфигурацию Nginx так, чтобы явно разделялись статический клиент и API backend.

**Требования:**

1. Настроены отдельные `location` (или эквивалент) для frontend (static) и для API backend.
2. Проксирование на backend использует имя сервиса Compose (не адрес только с хоста разработчика).
3. Заданы базовые таймауты проксирования (connect/send/read или согласованный минимум).
4. После перезагрузки конфигурации / пересоздания сервиса Nginx оба маршрута работают.

---

## Задание 2. Элементы балансировки и кэширования

**Описание.** Добавьте в конфигурацию хотя бы один элемент повышения устойчивости или эффективности публикации.

**Требования (выполнить не менее двух пунктов):**

1. **Балансировка или резерв:** upstream с двумя backend-инстансами *или* явная схема с `least_conn` / `weight` / запасным сервером (учебный минимум — два replica backend в Compose + upstream).
2. **Кэширование:** кэш статики и/или выбранных безопасных GET-ответов API (с понятными `proxy_cache_path` / заголовками) *либо* эквивалентная учебная схема кэша, согласованная с преподавателем.
3. **Ограничение нагрузки (альтернатива пункту 1 или 2):** `limit_req` / `limit_conn` на API *или* отдельный health-location для проверки живости backend.

В отчёте к защите кратко указать, какие два пункта реализованы и зачем.

---

## Задание 3. Multi-stage сборка образов

**Описание.** Оптимизируйте сборку хотя бы одного сервиса (frontend и/или backend) с помощью multi-stage Dockerfile.

**Требования:**

1. В Dockerfile не менее двух стадий (сборщик / runtime или build / serve).
2. Итоговый runtime-образ не содержит инструментов сборки, не нужных для запуска (для frontend — типично только артефакты static + сервер/nginx; для backend — приложение без кэша сборки лишних слоёв).
3. Образ успешно собирается; размер итогового образа меньше одностадийного варианта того же сервиса *или* студент обосновывает выигрыш (слои, отсутствие dev-зависимостей) на защите.
4. Контур Compose после замены образа поднимается и проходит проверку UI + API.

---

## Задание 4. Проверка кастомизированного контура

**Описание.** Подтвердите работоспособность доработанного production-контура.

**Требования:**

1. Выполнены `compose up` (или эквивалент) и проверка: UI открывается, ключевой API отвечает.
2. Продемонстрировано изменение поведения после кастомизации (например, запрос уходит на нужный `location`; кэш/лимит/второй инстанс — по выбранным пунктам задания 2).
3. Студент объясняет путь запроса и назначение директив Nginx / стадий Dockerfile.

---

## Порядок выполнения и сдачи

1. Исходить из контура занятия 3.4.
2. Доработать Nginx (задания 1–2) и Dockerfile (задание 3).
3. Пересобрать образы, перезапустить контур, продемонстрировать проверку.
4. Кратко обосновать выбранные элементы балансировки/кэша/ограничений.

**Формат сдачи:** обновлённые файлы Nginx и Dockerfile в репозитории проекта; устная защита с демонстрацией.

## Правила использования внешних материалов и генеративного ИИ

- Допускается использование официальной документации Nginx и Docker, а также учебных шаблонов преподавателя.
- Допускается применение генеративного ИИ для справок и черновиков конфигурации при условии понимания директив и способности изменить их на защите.
- Запрещается сдавать конфигурацию, которую студент не может объяснить.
- Секреты и ключи доступа в репозиторий не включаются.
