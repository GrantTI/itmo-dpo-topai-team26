# Лабораторная работа
## Тема: Конфигурирование обратного прокси-сервера и многостадийная сборка контейнерных образов

**Цель работы:** Сформировать умение кастомизировать контур развёртывания клиент-серверного приложения с моделями машинного обучения: уточнить маршрутизацию на обратном прокси-сервере Nginx, применить элементы балансировки нагрузки и кэширования, использовать многостадийную сборку контейнерных образов.

**Продолжительность:** 6 академических часов

**Связь с занятием:** 3.5

**Проверяемые результаты обучения**

| Компетенция | Индикатор | Уровень | Дескриптор в контексте задания |
|---|---|---|---|
| LC-5. Способен применять и (или) проектировать инструменты и инженерные практики промышленной разработки, развёртывания, эксплуатации и мониторинга систем ИИ | LC-5.1 | С | Кастомизирует обратный прокси-сервер и параметры контейнерной сборки для рабочей версии приложения; настраивает маршрутизацию и элементы устойчивости и производительности публикации сервисов |

**Необходимые условия:** выполненная лабораторная работа занятия 3.4 (базовый контур Compose с Nginx); Docker и Docker Compose; текстовый редактор.

**Форма контроля:** текущая (защита лабораторной работы).

---

## Задание 1. Уточнение маршрутизации на обратном прокси-сервере

**Описание.** Доработать конфигурацию Nginx таким образом, чтобы явно разделялись раздача статических ресурсов клиентской части и обращение к программному интерфейсу серверной части.

**Требования:**

1. Настроены отдельные блоки `location` (либо эквивалент) для статических ресурсов клиентской части и для программного интерфейса серверной части.
2. Проксирование на серверную часть использует имя сервиса Compose, а не исключительно адрес хоста разработки.
3. Заданы базовые значения времени ожидания проксирования (connect/send/read либо согласованный минимум).
4. После применения конфигурации оба маршрута функционируют корректно.

---

## Задание 2. Элементы балансировки нагрузки, кэширования и ограничения запросов

**Описание.** Дополнить конфигурацию элементами повышения устойчивости или эффективности публикации сервисов.

**Требования:** реализовать **два** пункта из приведённого перечня по следующей схеме и кратко обосновать выбор на защите:

- **обязательно** один пункт из позиций 1–2 (балансировка нагрузки / резервирование либо кэширование);
- **второй** пункт — любой из позиций 1–3.

1. **Балансировка нагрузки или резервирование:** блок `upstream` с двумя экземплярами серверной части либо схема с `least_conn` / `weight` / резервным сервером (учебный минимум — два экземпляра серверной части в Compose и соответствующий `upstream`).
2. **Кэширование:** кэширование статических ресурсов и/или выбранных безопасных ответов метода GET программного интерфейса (с настройкой `proxy_cache_path` и сопутствующих директив) либо эквивалентная учебная схема, согласованная с преподавателем.
3. **Ограничение интенсивности запросов или проверка состояния:** `limit_req` / `limit_conn` для программного интерфейса либо отдельный `location` для проверки состояния (health) серверной части.

---

## Задание 3. Многостадийная сборка контейнерных образов

**Описание.** Оптимизировать сборку не менее одного сервиса (клиентской и/или серверной части) с применением многостадийного Dockerfile.

**Требования:**

1. В Dockerfile предусмотрено не менее двух стадий (сборка и выполнение либо эквивалентная схема).
2. Итоговый образ выполнения не содержит инструментов сборки, не требуемых для запуска.
3. Образ успешно собирается; на защите приводится сравнение с одностадийным вариантом (уменьшение размера образа и/или исключение зависимостей среды разработки) либо иное обоснование выигрыша.
4. После замены образа контур Compose запускается и проходит проверку клиентского интерфейса и программного интерфейса.

---

## Задание 4. Проверка кастомизированного контура

**Описание.** Подтвердить работоспособность доработанного контура развёртывания.

**Требования:**

1. Выполнены сборка и запуск контура; клиентский интерфейс открывается, ключевой метод программного интерфейса отвечает.
2. Продемонстрировано изменение поведения после кастомизации (маршрутизация по `location`; эффекты выбранных двух пунктов задания 2).
3. Обучающийся поясняет путь запроса и назначение ключевых директив Nginx и стадий Dockerfile.

---

## Порядок выполнения и сдачи

1. Использовать в качестве основы контур занятия 3.4.
2. Доработать конфигурацию Nginx (задания 1–2) и Dockerfile (задание 3).
3. Пересобрать образы, перезапустить контур, продемонстрировать проверку.
4. Обосновать выбор двух элементов задания 2.

**Формат сдачи:** обновлённые файлы конфигурации Nginx и Dockerfile в репозитории проекта; устная защита с демонстрацией.

## Критерии оценивания

Максимум: **15 баллов**. Работа считается зачтённой при результате не ниже **9 баллов**.

| Критерий | Баллы | Показатели |
|---|:---:|---|
| Маршрутизация на обратном прокси-сервере | 0–4 | Разделение статики и программного интерфейса; проксирование на сервис Compose; заданы timeout |
| Два элемента устойчивости или эффективности | 0–4 | Выполнена схема «один из пунктов 1–2 + второй из 1–3»; поведение продемонстрировано |
| Многостадийная сборка | 0–4 | Не менее двух стадий; рабочий образ; контур запускается |
| Обоснование решения | 0–3 | Поясняются директивы, стадии сборки и путь запроса |

## Правила использования внешних материалов и генеративного ИИ

- Допускается использование официальной документации Nginx и Docker, а также учебных шаблонов, предоставленных преподавателем.
- Допускается применение генеративного ИИ для справок и подготовки черновиков конфигурации при условии понимания директив и способности изменить их на защите.
- Не допускается сдача конфигурации, которую обучающийся не может пояснить.
- Секреты и ключи доступа в репозиторий не включаются.
