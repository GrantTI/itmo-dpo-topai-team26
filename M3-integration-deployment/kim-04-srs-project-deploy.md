# Самостоятельная работа
## Тема: Развёртывание итогового приложения дисциплины в контейнерной среде

**Цель работы:** Закрепить навыки настройки и кастомизации контура развёртывания на материале собственного проекта дисциплины: подготовить контейнерные образы, описать оркестрацию сервисов и публикацию через Nginx, обеспечить воспроизводимый запуск и проверку работоспособности сервисов.

**Продолжительность:** 8 академических часов

**Связь с занятием:** 3.6

**Проверяемые результаты обучения**

| Компетенция | Индикатор | Уровень | Дескриптор в контексте задания |
|---|---|---|---|
| LC-5. Способен применять и (или) проектировать инструменты и инженерные практики промышленной разработки, развёртывания, эксплуатации и мониторинга систем ИИ | LC-5.1 | С | Самостоятельно настраивает и кастомизирует контур развёртывания своего клиент-серверного приложения с моделями машинного обучения; обеспечивает воспроизводимый запуск рабочей версии программного обеспечения и демонстрирует доступность сервисов |

**Необходимые условия:** выполненные лабораторные работы занятий 3.4 и 3.5 (либо эквивалентный контур); собственный проект дисциплины с серверной и клиентской частями; Docker и Docker Compose.

**Форма контроля:** рубежная (сдача контура развёртывания перед переходом к тестированию и защите в модуле 4).

**Соотношение с лабораторными работами 3.4–3.5.** Инкремент самостоятельной работы состоит в переносе освоенных практик на итоговый репозиторий обучающегося, обеспечении воспроизводимого запуска по письменной инструкции и подготовке описания контура для модуля 4. Повторная демонстрация того же учебного стенда без адаптации к собственному проекту не засчитывается.

**Соотношение с итоговой проектной работой.**

| Что оценивается | Занятие 3.6 (этот КИМ) | Критерий 6 проектной работы («промышленные практики…») и смежные |
|---|---|---|
| Воспроизводимый контур (Dockerfile, Compose, `.env.example`, Nginx, инструкция `up`/`down`) | **Да — основной объект оценки** | На защите проекта контур **предполагается уже сданным** в 3.6; критерий 6 не дублирует полную пересдачу Compose «с нуля», а учитывает наличие и использование практик развёртывания/сопровождения в составе продукта |
| Полнота ML-функциональности, качество UI, тесты, презентация | **Нет** | **Да** (критерии проекта / модуль 4) |
| Smoke-проверка доступности UI и API (в т.ч. метод, связанный с моделью) | **Да** (факт доступности после запуска) | Качество сценариев работы с моделью — в критериях проекта (в т.ч. критерий 5), не в 3.6 |

Повторное начисление полного балла за тот же контур и в 3.6, и в критерии 6 проекта не производится: в проекте фиксируется опора на уже принятый контур 3.6 (или его эволюция), а не вторая сдача той же лабораторной.

---

## Задание 1. Контур развёртывания собственного проекта

**Описание.** Перенести и адаптировать практики занятий 3.4–3.5 на репозиторий собственного проекта дисциплины.

**Требования:**

1. В проекте имеются Dockerfile для серверной и клиентской частей (допускается многостадийная сборка).
2. Описан файл `docker-compose.yml` с сетью сервисов; взаимодействие сервисов осуществляется по именам Compose.
3. Параметры среды вынесены в переменные окружения; в репозитории представлен файл `.env.example` без секретов.
4. Настроен Nginx для публикации клиентского интерфейса и/или программного интерфейса серверной части.

---

## Задание 2. Воспроизводимый запуск и проверка работоспособности

**Описание.** Обеспечить запуск контура по документированной процедуре и подтвердить работоспособность.

**Требования:**

1. Контур запускается по краткой инструкции (раздел развёртывания в README либо отдельный файл: сборка и запуск).
2. После запуска доступны клиентский интерфейс и метод программного интерфейса, связанный с моделью (минимальный smoke-тест — например, перечень моделей; «только health» без обращения к модели недостаточно).
3. Выполняются штатные останов и повторный запуск без недокументированного ручного вмешательства в работу контейнеров.
4. Секреты отсутствуют в сдаваемых файлах и в истории репозитория.

---

## Задание 3. Описание состава контура развёртывания

**Описание.** Зафиксировать архитектуру развёртывания для последующего использования в модуле 4.

**Требования:**

1. Приведён перечень сервисов Compose и их назначение (серверная часть, клиентская часть, Nginx; иные сервисы — при наличии в проекте).
2. Указаны порты публикации и основные переменные окружения (без значений секретов).
3. Описан путь запроса: клиент → Nginx → серверная часть → применение модели (внутрипроцессно в серверной части либо иной согласованный в проекте способ).

---

## Порядок выполнения и сдачи

1. Адаптировать средства контейнеризации под собственный проект.
2. Запустить контур и выполнить проверку клиентского интерфейса и программного интерфейса.
3. Подготовить краткое описание состава контура.
4. Представить материалы преподавателю в рамках рубежного контроля модуля 3.

**Формат сдачи:** репозиторий проекта с файлами развёртывания; демонстрация запуска; краткое описание контура (в README либо в отдельном файле).

---

## Критерии оценивания

Максимум: **15 баллов**. Работа считается зачтённой при результате не ниже **9 баллов**.

| Критерий | Баллы | Показатели |
|---|:---:|---|
| Полнота контура | 0–5 | Dockerfile, Compose, пример файла окружения, Nginx для собственного проекта |
| Воспроизводимый запуск | 0–5 | Запуск и останов; доступность UI и метода API, связанного с моделью (smoke); инструкция по запуску |
| Описание контура | 0–3 | Сервисы, порты, путь запроса |
| Корректность конфигурации | 0–2 | Отсутствие секретов в системе контроля версий; ясная структура файлов |

*В рамках данной самостоятельной работы не оцениваются:* качество обучения моделей, оформление клиентского интерфейса, покрытие автоматизированными тестами, презентация к защите проекта.

- Допускается использование собственных материалов лабораторных работ занятий 3.4–3.5 и официальной документации Docker и Nginx.
- Допускается применение генеративного ИИ для справки при условии понимания конфигурации и способности воспроизвести её на сдаче.
