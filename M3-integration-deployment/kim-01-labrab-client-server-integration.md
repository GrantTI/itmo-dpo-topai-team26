# Лабораторная работа
## Тема: Интеграция клиентской и серверной частей приложения с применением модели машинного обучения

**Цель работы:** Подтвердить работоспособность связки клиентской части (модуль 2) и серверной части (модуль 1): согласовать контракт REST API и доступ браузера к API, продемонстрировать из пользовательского интерфейса сценарий с применением модели через API серверной части.

**Продолжительность:** 2 академических часа

**Связь с занятием:** 3.3 (с опорой на содержание занятий 3.1–3.2)

**Место в траектории модуля.** Занятие 3.3 — **приёмка интеграции** перед контейнерным контуром (3.4–3.5). Настройка CORS и/или dev-proxy, уже выполненная в модуле 2, засчитывается при демонстрации; новым обязательным результатом является сквозной сценарий **из UI** с обращением к функциональности модели.

**Проверяемые результаты обучения**

| Компетенция | Индикатор | Уровень | Дескриптор в контексте задания |
|---|---|---|---|
| PL-1. Способен применять язык программирования Python для решения задач в области ИИ | PL-1.1 | С | При необходимости донастраивает на FastAPI параметры доступа клиента (политика CORS и/или конфигурация базового URL API) и обеспечивает согласованный контракт ответов для клиентской части |
| LC-5. Способен применять и (или) проектировать инструменты и инженерные практики промышленной разработки, развёртывания, эксплуатации и мониторинга систем ИИ | LC-5.1 | С | Организует HTTP-взаимодействие клиента и сервера в интегрированном учебном контуре и демонстрирует вызов и применение модели машинного обучения через API серверной части из пользовательского интерфейса |

**Необходимые условия:** выполненные серверная часть модуля 1 (FastAPI с API моделей) и клиентская часть модуля 2 с экранами/сценариями обращения к API; текстовый редактор; среда выполнения Python 3.x и среда запуска клиентской части (Node.js / Vite либо эквивалент стека модуля 2). Рекомендуется сданное эссе к занятиям 3.1–3.2 (либо готовность кратко сформулировать выбранный паттерн in-process на защите).

**Форма контроля:** текущая (защита лабораторной работы).

## Задание 1. Согласование контракта и базового адреса API

**Описание.** Зафиксировать, как клиентская часть обращается к серверной: базовый URL, префикс путей, ключевые методы для проверки интеграции.

**Требования:**

1. В клиентской части базовый адрес API задаётся через переменные окружения (или иной принятый в стеке модуля 2 механизм конфигурации) и не «зашит» произвольно в нескольких компонентах без единой точки настройки.
2. В репозитории представлен пример конфигурации без секретов (например, `.env.example` для клиента и/или сервера).
3. Кратко перечислены методы, используемые для демонстрации (не менее двух из числа: проверка состояния сервиса; перечень моделей; обучение; прогнозирование — в зависимости от готовности проекта).

---

## Задание 2. Доступ браузера к API (CORS и/или прокси разработки)

**Описание.** Обеспечить работоспособность запросов из клиентской части к серверной при типичной схеме локальной разработки (разные порты origin).

**Требования:**

1. Реализован **хотя бы один** согласованный способ:
   - настройка политики CORS на стороне FastAPI; и/или
   - проксирование запросов к API средствами инструмента сборки клиентской части (dev-proxy) либо иным документированным способом.
2. Если CORS и/или dev-proxy уже настроены в модуле 2 и запросы из UI к API проходят — достаточно подтвердить это на защите и пояснить выбранный способ; повторная «с нуля» настройка не требуется.
3. На защите обучающийся поясняет ограничения выбранного способа относительно последующей публикации через Nginx (занятия 3.4–3.5).
4. При использовании CORS параметры разрешённых источников по возможности выносятся в конфигурацию окружения.

---

## Задание 3. Сквозная проверка с применением модели из пользовательского интерфейса

**Описание.** Подтвердить, что контур «UI → сервер → модель» работоспособен.

**Требования:**

1. **Обязательно** демонстрируется успешный сценарий **из клиентского интерфейса** модуля 2, включающий обращение к функциональности модели модуля 1 (перечень моделей и/или обучение и/или прогнозирование). Проверка через OpenAPI/Swagger, `curl` или `httpx` допускается только как **дополнение**, но не заменяет демонстрацию из UI.
2. При недоступности серверной части (сервер остановлен) клиентская часть ведёт себя предсказуемо: отображает понятное сообщение об ошибке и/или не завершает работу вкладки/приложения аварийно; процесс сервера при этом не рассматривается как объект «обработки» со стороны клиента.
3. На защите кратко соотносится результат с паттерном размещения инференса (для учебного стенда — внутрипроцессный вызов в backend; при наличии эссе — со ссылкой на выбор в эссе).

---

## Порядок выполнения и сдачи

1. Согласовать базовый URL и перечень демонстрационных методов API.
2. Подтвердить или донастроить CORS и/или прокси разработки.
3. Запустить серверную и клиентскую части; продемонстрировать сквозной сценарий **из UI** с моделью.
4. Пояснить выбор способа интеграции относительно публикации через reverse-proxy.

**Формат сдачи:** работоспособный исходный код в репозитории проекта обучающегося; устная защита с демонстрацией UI; файлы примеров окружения без секретов.

## Критерии оценивания

Максимум: **10 баллов**. Работа считается зачтённой при результате не ниже **6 баллов**.

| Критерий | Баллы | Показатели |
|---|:---:|---|
| Контракт и конфигурация адреса API | 0–2 | Единая настройка базового URL; `.env.example` (или аналог); перечень демонстрационных методов |
| CORS и/или прокси разработки | 0–2 | Запросы из UI к API работают; способ пояснён (в том числе если унаследован из модуля 2) |
| Сквозной сценарий с моделью из UI | 0–4 | Из интерфейса продемонстрирован вызов функциональности модели через API модуля 1 |
| Устойчивость клиента и обоснование | 0–2 | Предсказуемая реакция UI при недоступности API; связь с 3.1 и паттерном in-process; понимание перехода к Nginx |

## Правила использования внешних материалов и генеративного ИИ

- Допускается использование официальной документации FastAPI (CORSMiddleware), стека клиентской части модуля 2 и материалов занятий 3.1–3.2.
- Допускается применение генеративного ИИ для справок и подготовки черновиков конфигурации при условии, что обучающийся понимает решение и способен воспроизвести его логику на защите.
- Не допускается представление непроверенного сгенерированного кода без возможности пояснить назначение фрагментов.
- Секреты и ключи доступа в репозиторий и отчётные материалы не включаются.
